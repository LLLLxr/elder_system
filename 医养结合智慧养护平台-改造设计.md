# 医养结合智慧养护平台 - 改造设计

## 一、改造原则

- 保留现有 **前端双应用 + 后端微服务** 架构
- 不做整体推翻，按业务域增量扩展
- 流程统一收敛到 `careorchestration`
- 三端共用数据结构放到 `common-lib`
- 共用 UI 组件优先复用 `packages/common-react`

---

## 二、总体架构

### 1. 前端
- **医护端**：复用 `smart-nursing-home-demo`
- **护理员端 / 家属端**：复用 `smart-nursing-home-service-portal`
- 当前阶段先按 Web/H5 形态承载，后续再决定是否单独拆原生 APP

### 2. 后端
按业务域继续扩展：
- `admission`：入院预约、评估、签约入口
- `health`：健康评估、查房、健康档案
- `contract`：云签字、合同流转
- `caredelivery`：护理执行、服务清单、打卡
- `quality`：培训、考核、满意度、评价
- `careorchestration`：统一流程状态、待办、审核、回退、日志
- `ops`：预警、设备告警、视频/通知

---

## 三、推荐业务拆分

### 1. 医护端
建议包含：
- 人员管理
- 护理员资质审核
- 入院评估
- 护理记录
- 查房记录
- 预约审核
- 云签字
- 告警待办
- 统计看板

### 2. 护理员端
建议包含：
- 注册与资质上传
- 每日打卡
- 服务清单执行
- 培训视频
- 考核结果
- 云探视入口

### 3. 家属端
建议包含：
- 机构信息展示
- 预约参观
- 云签字
- 服务记录查看
- 健康评估 / 查房记录查看
- 月度满意度
- 云探视入口

---

## 四、流程设计

### 1. 入院闭环
1. 家属预约参观
2. 医护端审核
3. 云签字
4. 入院健康评估
5. 判定失能等级
6. 制定服务清单

### 2. 日常养护闭环
1. 护士 / 医生每日记录
2. 护理员打卡执行
3. 家属实时查看
4. 月度满意度评价

### 3. 护理员管理闭环
1. 护理员注册并上传资质
2. 医护端审核通过
3. 线上培训
4. 季度考核
5. 日常考评

### 4. 预警闭环
1. 手环 / 设备产生预警
2. 进入待办
3. 医护端处理
4. 记录处理结果与日志

## 五、状态与流程中心

建议继续复用现有 `careorchestration` 的状态机模式，把以下流程纳入统一管理：

- 预约参观审核
- 护理员资质审核
- 合同签署完成
- 入院评估完成
- 每日护理任务完成情况
- 月度满意度 / 季度考核
- 风险预警待处理

## 九、验证方式

### 前端
- 启动两个前端应用
- 检查路由、登录、权限、菜单
- 验证三端核心页面是否可进入

### 后端
- 执行 `mvn test`
- 检查新 API 路由前缀
- 校验角色权限与状态流转

### 业务
- 走完整的“预约参观 → 签约 → 评估 → 在服 → 评价”流程
- 走完整的“注册 → 审核 → 培训 → 考核”流程
- 校验家属端可见信息与医护端录入一致
