# 医养结合智慧养护平台 - 阶段一后端实施清单

> 范围：阶段一只覆盖“家属预约参观”和“护理员资质审核”两个闭环，以及最小工作流支撑。

---

# 一、后端实施目标

阶段一后端目标：

1. 完成角色与权限补齐
2. 完成预约参观 API 闭环
3. 完成护理员资质申请/审核 API 闭环
4. 完成最小工作流待办与日志能力
5. 保证前后端可联调

---

# 二、模块实施分工

## 1. iam-service
负责：
- 新角色
- 新权限
- 登录态返回角色/权限

### 任务
- 新增角色：
  - `SUPER_ADMIN`
  - `DOCTOR`
  - `NURSE`
  - `CAREGIVER`
  - `FAMILY_MEMBER`
- 新增权限：
  - `admission:reservation:review`
  - `quality:caregiver-qualification:review`
  - `admission:reservation:create`
  - `quality:caregiver-qualification:create`
- 更新初始化 SQL / 初始化逻辑
- 验证角色权限查询接口返回正确

验收标准：
- 新角色可分配
- 登录后能拿到对应权限集合

---

## 2. admission 模块
负责：
- 家属预约参观
- 医护端审核预约

### 任务 2.1：数据层
- 新增 `family_visit_reservation` 对应 PO
- 新增 Repository
- 支持按状态、日期、联系人分页查询
- 支持按 ID 查询
- 支持按家属用户查询我的预约记录

### 任务 2.2：领域模型
- 新增预约领域模型对象
- 封装状态流转：
  - 创建预约 → `PENDING`
  - 审核通过 → `APPROVED`
  - 审核拒绝 → `REJECTED`

### 任务 2.3：服务层
- 查询可预约时段
- 创建预约申请
- 查询我的预约记录
- 分页查询审核列表
- 查询预约详情
- 审核通过
- 审核拒绝

### 任务 2.4：控制器
新增接口：
- `GET /admission/family-visit-slots`
- `POST /admission/family-visit-reservations`
- `GET /admission/family-visit-reservations/my`
- `GET /admission/family-visit-reservations`
- `GET /admission/family-visit-reservations/{id}`
- `POST /admission/family-visit-reservations/{id}/approve`
- `POST /admission/family-visit-reservations/{id}/reject`

### 任务 2.5：校验规则
- 预约日期不能为空
- 时间段不能为空
- 联系人姓名不能为空
- 联系电话不能为空
- 同时段容量校验（阶段一可先固定 1 人）

验收标准：
- 家属端可创建预约
- 医护端可审核预约
- 状态能正确流转

---

## 3. quality 模块
负责：
- 护理员资质提交
- 医护端资质审核

### 任务 3.1：数据层
- 新增 `caregiver_qualification_application` 对应 PO
- 新增 Repository
- 支持按状态、姓名分页查询
- 支持按 ID 查询
- 支持按 userId 查询最近申请

### 任务 3.2：领域模型
- 新增资质申请领域模型对象
- 封装状态流转：
  - 创建申请 → `PENDING`
  - 审核通过 → `APPROVED`
  - 审核拒绝 → `REJECTED`

### 任务 3.3：服务层
- 提交资质申请
- 查询我的申请状态
- 分页查询审核列表
- 查询申请详情
- 审核通过
- 审核拒绝

### 任务 3.4：控制器
新增接口：
- `POST /quality/caregiver-qualification-applications`
- `GET /quality/caregiver-qualification-applications/my`
- `GET /quality/caregiver-qualification-applications`
- `GET /quality/caregiver-qualification-applications/{id}`
- `POST /quality/caregiver-qualification-applications/{id}/approve`
- `POST /quality/caregiver-qualification-applications/{id}/reject`

### 任务 3.5：校验规则
- 护理员姓名不能为空
- 身份证号不能为空
- 身份证图片不能为空
- 护理员证图片不能为空
- 若已有待审核申请，是否禁止重复提交（阶段一建议后端做限制）

验收标准：
- 护理员可提交申请
- 医护端可审核
- 审核结果可回流查询

---

## 4. careorchestration 模块
负责：
- 最小待办任务
- 最小流转日志

### 任务 4.1：待办任务复用或扩展
优先检查是否直接复用现有：
- `ServiceJourneyTaskPo`
- `ServiceJourneyTaskService`
- `ServiceJourneyTransitionLogPo`
- `ServiceJourneyTransitionLogService`

如果直接复用成本高，再补最小 workflow 结构。

### 任务 4.2：任务创建时机
- 预约提交成功后创建 `FAMILY_VISIT_RESERVATION_REVIEW` 待办
- 资质申请提交成功后创建 `CAREGIVER_QUALIFICATION_REVIEW` 待办

### 任务 4.3：任务完成时机
- 预约审核通过/拒绝后关闭待办
- 资质审核通过/拒绝后关闭待办

### 任务 4.4：流转日志
记录：
- 创建
- 审核通过
- 审核拒绝

验收标准：
- 每个业务都能看到对应待办
- 审核后待办和日志同步更新

# 五、阶段一后端完成标准

满足以下条件即可视为阶段一后端完成：

1. 家属预约参观闭环可跑通
2. 护理员资质申请/审核闭环可跑通
3. 角色权限可区分
4. 待办和日志最小闭环存在
5. 接口可供前端直接联调
