# 医养结合智慧养护平台 - 阶段五表结构草案

> 范围：设备预警、风险待办、云探视、通知。

---

# 一、建表原则

- 阶段五主要落在 `ops` 与 `careorchestration`
- 预警与通话记录建议单独落表，便于审计和查询
- 通知表建议独立，方便三端统一消费

---

# 二、ops 域表结构

## 1. device_alert_record
用途：设备 / 手环预警记录

### 建议字段
- `id` bigint PK
- `elder_id` bigint NOT NULL
- `device_id` varchar(64) NOT NULL
- `alert_type` varchar(64) NOT NULL
- `alert_level` varchar(32) NOT NULL
- `alert_content` varchar(255) NOT NULL
- `alert_status` varchar(32) NOT NULL
- `triggered_at` datetime NOT NULL
- `handled_by` varchar(64) NULL
- `handled_at` datetime NULL
- `handle_result` varchar(255) NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(elder_id, triggered_at desc)`
- `(alert_status, triggered_at desc)`
- `(alert_level, triggered_at desc)`

---

## 2. remote_visit_call_record
用途：云探视通话记录

### 建议字段
- `id` bigint PK
- `elder_id` bigint NOT NULL
- `family_user_id` bigint NOT NULL
- `requested_by` varchar(32) NOT NULL
- `accepted_by` varchar(64) NULL
- `rejected_by` varchar(64) NULL
- `reason` varchar(255) NULL
- `call_status` varchar(32) NOT NULL
- `started_at` datetime NULL
- `ended_at` datetime NULL
- `duration_seconds` int NULL
- `record_url` varchar(255) NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(elder_id, created_at desc)`
- `(family_user_id, created_at desc)`
- `(call_status, created_at desc)`

---

## 3. notification_record
用途：统一通知表

### 建议字段
- `id` bigint PK
- `receiver_user_id` bigint NOT NULL
- `receiver_role` varchar(64) NOT NULL
- `notification_type` varchar(64) NOT NULL
- `title` varchar(128) NOT NULL
- `content` varchar(500) NOT NULL
- `read_status` varchar(32) NOT NULL
- `biz_type` varchar(64) NULL
- `biz_id` bigint NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(receiver_user_id, read_status, created_at desc)`
- `(receiver_role, read_status)`
- `(biz_type, biz_id)`

---

# 四、业务关系说明

## 1. 设备预警
- `device_alert_record` 记录预警本身
- orchestration 记录待办与处理流转
- notification 可同步发提醒给相关医护角色

## 2. 云探视
- `remote_visit_call_record` 记录一条通话请求及其生命周期
- 如需要推送，也可以在 notification 中写一条记录

## 3. 通知
- `notification_record` 作为统一通知承载表
- 三端都从这张表读取自己的通知列表

# 五、阶段五建表优先级

## P0
1. `device_alert_record`
2. `remote_visit_call_record`
3. `notification_record`

## P1
1. 预警规则配置表
2. 通话录制索引表
3. 通知模板表

# 六、后续如需增强的表结构方向

## 1. 预警规则配置
如果要后台可配置预警规则，建议补：
- `alert_rule_config`

## 2. 通知模板
如果通知内容需要标准化，建议补：
- `notification_template`

## 3. 通话录制
如果云探视要接入录制，建议补：
- `remote_visit_call_recording`

# 七、阶段五仍待确认的表结构问题

1. 通知是统一表还是分端独立表
2. 云探视是否需要保存录制链接
3. 预警是否需要分配给具体医护人员
4. 预警状态是否需要更多中间态
5. 远程探视请求是否允许排队
