# 医养结合智慧养护平台 - 阶段三表结构草案

> 范围：护理员每日打卡执行、护士护理记录、医生查房记录、家属查看在服信息。

---

# 一、建表原则

- 阶段三主要落在 `caredelivery` 和 `health`
- 保持“主记录 + JSON 明细”的轻量设计，先跑通业务
- 后续如统计和扩展需求变多，再把 JSON 拆成子表

---

# 二、caredelivery 域表结构

## 1. caregiver_check_in_record
用途：护理员每日打卡执行记录

### 建议字段
- `id` bigint PK
- `elder_id` bigint NOT NULL
- `caregiver_id` bigint NOT NULL
- `service_plan_id` bigint NOT NULL
- `task_date` date NOT NULL
- `task_items_json` text NOT NULL
- `completion_status` varchar(32) NOT NULL
- `completion_time` datetime NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(caregiver_id, task_date desc)`
- `(elder_id, task_date desc)`
- `(service_plan_id, task_date)`

### 建议约束
- 同一护理员 + 同一老人 + 同一天 + 同一服务清单，阶段三建议只保留一条有效记录

---

## 2. nurse_care_record
用途：护士每日护理记录

### 建议字段
- `id` bigint PK
- `elder_id` bigint NOT NULL
- `nurse_id` bigint NOT NULL
- `service_plan_id` bigint NULL
- `record_date` date NOT NULL
- `record_form_data` text NOT NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(elder_id, record_date desc)`
- `(nurse_id, record_date desc)`

### 建议约束
- 同一老人同一天允许一条主护理记录（阶段三可先在 service 层控制）

---

## 3. elder_service_plan
阶段三主要复用阶段二的 `elder_service_plan`，不一定新增字段。

可选扩展字段：
- `assigned_caregiver_id` bigint NULL

如果后续要固定某位护理员负责某位老人，可补这个字段；否则阶段三可以先不加。

---

# 三、health 域表结构

## 1. doctor_round_record
用途：医生每日查房记录

### 建议字段
- `id` bigint PK
- `elder_id` bigint NOT NULL
- `doctor_id` bigint NOT NULL
- `content` varchar(2000) NOT NULL
- `risk_flag` boolean NOT NULL
- `round_time` datetime NOT NULL
- `created_at` datetime NOT NULL
- `updated_at` datetime NOT NULL

### 建议索引
- `(elder_id, round_time desc)`
- `(doctor_id, round_time desc)`
- `(risk_flag, round_time desc)`

---

# 四、orchestration 扩展建议

如继续复用现有工作流任务表和日志表，阶段三只扩展枚举与写入逻辑。

### 新增 taskType
- `DAILY_CARE_TASK_EXECUTION`
- `NURSE_CARE_RECORD`
- `DOCTOR_ROUND_RECORD`

### 新增日志 event
- `CARE_TASK_CHECKED_IN`
- `NURSE_CARE_RECORD_CREATED`
- `NURSE_CARE_RECORD_UPDATED`
- `DOCTOR_ROUND_RECORD_CREATED`
- `DOCTOR_ROUND_RECORD_UPDATED`

---

# 五、业务关系说明

## 1. 每日服务执行
- 来源：`elder_service_plan`
- 结果：`caregiver_check_in_record`
- 家属端查看执行情况时，以打卡记录为准

## 2. 护士护理记录
- 与老人强关联
- 可关联当前服务清单，但不强依赖
- 家属端查看时只读展示

## 3. 医生查房记录
- 与老人强关联
- 记录日常健康观察与风险情况
- 家属端查看时只读展示

---

# 六、阶段三建表优先级

## P0
1. `caregiver_check_in_record`
2. `nurse_care_record`
3. `doctor_round_record`

## P1
1. 每日任务快照表（如果后续不满足动态生成）
2. 护理记录模板表
3. 查房记录模板表

---

# 七、如果要进一步增强的表结构方向

## 1. 每日任务快照表（后续可选）
若后续需要“任务分配、部分完成、多次提交、精确追责”，建议补：
- `daily_care_task`

用于提前生成每位护理员、每位老人的每日任务快照。

## 2. 服务清单项子表（后续可选）
若后续要做统计和报表，建议将 `plan_items_json` 拆分为：
- `elder_service_plan_item`

## 3. 打卡明细子表（后续可选）
若后续要单项分析每个服务项执行情况，建议将 `task_items_json` 拆分为：
- `caregiver_check_in_item`

---

# 八、阶段三仍待确认的表结构问题

1. 打卡记录是否允许一天多次提交
2. 护理记录是否要支持草稿状态
3. 查房记录是否需要风险等级而不只是布尔值
4. 是否要记录家属查看日志
5. 是否需要为家属端展示内容做脱敏字段控制
